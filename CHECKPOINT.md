## 1. Что за инструмент

Server Log Analyzer — веб-приложение для анализа access-логов веб-серверов (Nginx, Apache, combined log format). Главная цель — дать SEO-специалисту удобные метрики по crawl-бюджету, поисковым ботам, LLM-краулерам, ошибкам у ботов и поведению пользователей.

**Ключевая архитектурная идея:** весь парсинг происходит в браузере. Файл не уходит на сервер. Это решает две проблемы:

- **Приватность** — логи содержат IP адреса и User-Agent, их не хочется отправлять третьей стороне.
- **Масштабирование** — не нужно платить за серверные ресурсы, любой пользователь тянет свой файл своим процессором.

**Как это технически достижимо:**

- Файл читается потоково через `File.stream()` чанками по 8 МБ.
- Парсинг и агрегация идут в Web Worker, чтобы не блокировать UI.
- Агрегация идёт «на лету» — не храним все миллионы записей, а сразу считаем счётчики и топы.
- На выходе только готовые сводки: «столько-то запросов Googlebot», «этот URL видит 404 от ClaudeBot 5 раз» и т.п.

Протестировано на файлах до 200 МБ (более 1 млн строк) — обрабатывается за несколько секунд.

### Checkpoint 7 — Верификация ботов через DNS-over-HTTPS

- **Коммит:** `ff01442` — bot authenticity verification via DNS-over-HTTPS
- **Зачем:** любой парсер может представиться Googlebot или YandexBot. Без проверки подлинности все отчёты врут.
- **Что сделано:**
  - `client/src/lib/bot-verifier.ts` — DoH-клиент (Cloudflare `cloudflare-dns.com/dns-query`, JSON API) с параллелизмом 8 и кешем на сессию.
  - Worker собирает уникальные IP по каждому поисковому боту (`Googlebot*`, `Yandex*` и др.).
  - Проверка по алгоритму, рекомендованному Google и Яндексом:
    1. Reverse DNS lookup: IP → PTR-запись (имя хоста).
    2. Проверка суффикса: Google → `.googlebot.com`/`.google.com`/`.googleusercontent.com`, Yandex → `.yandex.ru`/`.yandex.net`/`.yandex.com`.
    3. Forward DNS lookup: имя хоста → IP.
    4. Сверка, что полученный IP совпадает с исходным.
  - Новый компонент `BotVerifyBadge.tsx` — иконки ✓/✗/⚠/? рядом с именем бота во всех таблицах.
  - Новая вкладка «Верификация»: объяснение метода, сводка (настоящие / поддельные / без PTR / ошибки сети), разбивка по ботам, отдельная таблица поддельных IP с PTR и причиной. Экспорт CSV.
  - Баннер над табами: CTA «Проверить» → прогресс → итог.
  - Из браузера наружу уходят только IP для DNS-запросов. Логи, URL, User-Agent остаются локально.

### 3.3 Web Worker — как работает агрегатор

`client/src/lib/log-worker.ts`, класс `StreamingAggregator`. Ключевая идея — на каждую строку лога обновляем ~10 Map/Set, но никогда не храним саму строку.

**Основные карты (Map) внутри агрегатора:**

| Карта | Что хранит | Использование |
|-------|-----------|---------------|
| `statusMap` | код → счётчик | Таблица «Детальные коды» |
| `uaMap` | UA-строка → {count, statusCounts, urlCounts} | Вкладка User-Agents |
| `botMap` | имя бота → {count, urls Set, errors} | Вкладка «Краулинг» |
| `urlMap` | URL → {count, statusSum} | Топ URL |
| `hourMap` | час → {total, bots} | Почасовой график |
| `statusByBotMap` | бот → (код → count) | Таблица кодов по ботам |
| `dailyBotMap` | дата → (бот → count) | Дневной стек-график |
| `detailMap` | `uaIdx|urlIdx|code` → count | Вкладка «Детализация» |
| `botErrorMap` | `бот\x01URL\x01код` → count | Вкладка «Ошибки ботов» |

**Ограничения (защита от OOM на 1 ГБ файлах):**

- `MAX_UA = 50 000` — уникальных UA
- `MAX_URLS = 200 000` — уникальных URL
- `MAX_URLS_PER_UA = 200` — URL на один UA
- `MAX_DETAIL_ROWS = 300 000` — строк детализации
- `MAX_BOT_ERRORS = 50 000` — комбинаций ошибок ботов

Если лимит достигнут, в UI появляется предупреждение «Данные урезаны».

### 3.7 Верификация ботов (DoH)

Ключевая логика — в `client/src/lib/bot-verifier.ts`. Запускается по кнопке:

1. Worker собирает уникальные IP по каждому поисковому боту (до 200 самых активных на бот).
2. Для каждого IP делается DoH-запрос к Cloudflare (`cloudflare-dns.com/dns-query`).
3. Результат кешируется на сессию, параллелизм 8 запросов.
4. В UI появляются иконки: ✓ подлинный, ✗ подделка, ⚠ смешанный, ? не проверяли.

**Важно:** browser-клиент не может делать DNS-запросы напрямую, поэтому используется DoH — DNS, завёрнутый в HTTPS. Из логов наружу уходят только IP-адреса. Всё остальное остаётся в браузере.

**Константа `VERIFIABLE_BOTS`** в `bot-verifier.ts` — список ботов, для которых возможна проверка. Сейчас это все варианты Google и Yandex. Для Bing и других подобной DNS-инструкции нет.

### 3.6 Формат логов

Поддерживается только **combined log format**:

```
127.0.0.1 - - [27/Mar/2026:10:15:30 +0300] "GET /page HTTP/1.1" 200 1234 "http://referer.com" "Mozilla/5.0 ..."
```

Это дефолт для Nginx и Apache. Если у пользователя другой формат — нужна предварительная конвертация.

Regex в `log-worker.ts`:

```js
/^(\S+)\s+\S+\s+\S+\s+\[([^\]]+)\]\s+"(\S+)\s+(\S+)\s*(\S*)"\s+(\d{3})\s+(\d+|-)\s+"([^"]*)"\s+"([^"]*)"/
```

## 6. Что делать, если что-то сломалось

### Дашборд пустой после загрузки файла

- Проверить формат лога — должен быть combined log. Первая строка файла должна начинаться с IP адреса.
- Открыть DevTools → Console. Worker пишет туда ошибки парсинга.

### Браузер зависает при загрузке большого файла

- Проверить размер — лимит ~1 ГБ.
- Если надо больше — увеличить `MAX_DETAIL_ROWS` и другие константы в `log-worker.ts`. Но это рискованно — браузер может упасть по памяти.

### Какой-то бот не определяется

- Открыть `client/src/lib/log-worker.ts`, найти `BOT_PATTERNS`.
- Добавить новый паттерн — более специфичные выше, общие ниже.
- Если бот должен быть на дневном графике — добавить его в `TRACKED_BOTS` и дать цвет в `BOT_COLORS` в `dashboard.tsx`.

### CSV не открывается в Excel

- Файл уже с BOM, должно работать. Если нет — в Excel «Данные → Из текста/CSV» и выбрать UTF-8.

- **Глубина URL** — распределение по уровням вложенности (`/category/` vs `/category/item/slug`).
- **Экспорт PDF-отчёта** одной кнопкой со всеми графиками.
- **Сравнение нескольких сессий** — например, логи этого месяца vs прошлого.
- **Детектор «забытых» URL** — важные страницы, которые боты не посещают.
- **Автоматическая блокировка поддельных IP** — после верификации сгенерировать готовый фрагмент для nginx `deny` или iptables.
- **Верификация Bing/Apple/DuckDuckGo** — у них тоже есть IP-листы или DNS-правила, можно расширить метод.
